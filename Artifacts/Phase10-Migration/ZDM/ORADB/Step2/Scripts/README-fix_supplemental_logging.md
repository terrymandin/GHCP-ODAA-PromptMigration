# README: fix_supplemental_logging.sh

## Purpose
Enables ALL COLUMNS and PRIMARY KEY supplemental logging on the source Oracle 12.2 database (`ORADB1`) — resolving **Issue 2 (Blocker)** from the ORADB Discovery Summary. This is a mandatory pre-requisite for ZDM `ONLINE_PHYSICAL` migration.

---

## Target Server
**ZDM Server** — run as `zdmuser` on `tm-vm-odaa-oracle-jumpbox` (10.1.0.8).  
The script connects to the **source server** (`tm-oracle-iaas`, 10.1.0.11) via SSH and executes SQL as `oracle` using `sudo`.

---

## Prerequisites

| Requirement | Detail |
|-------------|--------|
| Run as | `zdmuser` on the ZDM server |
| SSH key | `~/.ssh/iaas.pem` must exist and have permissions `600` |
| Source reachable | Port 22 open from ZDM server (10.1.0.8) to source (10.1.0.11) |
| Source DB running | `oradb` CDB instance must be `READ WRITE` |
| PDB1 open | Issue 1 (`fix_open_pdb1.sh`) must be resolved first |
| Archive logging | Database must be in ARCHIVELOG mode (confirmed ✅ in discovery) |

## What It Does

1. **User guard** — Exits immediately if not running as `zdmuser`.
2. **SSH connectivity test** — Verifies the ZDM server can reach the source server over SSH.
3. **Check current state** — Queries `V$DATABASE` to show the current supplemental logging flags (`LOG_DATA_MIN`, `LOG_DATA_PK`, `LOG_DATA_ALL`, etc.).
4. **Enable ALL COLUMNS logging** — Runs `ALTER DATABASE ADD SUPPLEMENTAL LOG DATA (ALL) COLUMNS;`.
5. **Enable PRIMARY KEY logging** — Runs `ALTER DATABASE ADD SUPPLEMENTAL LOG DATA (PRIMARY KEY) COLUMNS;`.
6. **Switch logfile** — Runs `ALTER SYSTEM SWITCH LOGFILE;` to immediately begin writing redo with the new supplemental logging level.
7. **Verify** — Re-queries `V$DATABASE` and confirms `SUPPLEMENTAL_LOG_DATA_ALL = YES`. Exits with code `1` if verification fails.

> SQL is passed to the remote session via base64 encoding to prevent shell quoting conflicts with single-quoted SQL values.

## Performance Impact

Enabling ALL COLUMNS supplemental logging increases the volume of redo log data generated by DML operations. For this database (2.08 GB, primarily system schemas with minimal user DML), the impact is expected to be negligible. On heavily-loaded production systems, monitor redo log generation rate and archive log destination size.

---

## Rollback / Undo

To remove ALL COLUMNS supplemental logging after migration is complete and verified:

```sql
-- As sysdba on source (after migration confirmed):
ALTER DATABASE DROP SUPPLEMENTAL LOG DATA (ALL) COLUMNS;
-- Note: Minimal and FK supplemental logging will remain.
```

> **Warning:** Do NOT remove supplemental logging while the ZDM migration job is active. Doing so will cause redo apply failures at the target and will abort the migration.
